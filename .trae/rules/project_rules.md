# Rule 1: Always Provide Ethiopian Context
Every prompt must include Ethiopian market context:
- Connectivity challenges (intermittent internet, low bandwidth)
- Device limitations (basic smartphones, feature phones)
- Language diversity (English, Amharic, Oromo)
- Literacy considerations
- Cultural practices around livestock
- Economic constraints of small-scale farmers

# Rule 2: Specify Offline-First Requirements
All features must work offline first:
- Local data storage for user information
- Caching of frequently accessed data
- Synchronization when internet is available
- Clear online/offline status indicators
- Data conflict resolution for offline edits
- Handle synchronization errors gracefully
- Provide clear error messages and resolutions
- Offer options to resolve synchronization conflicts

Always include in prompts:
- "Ensure this works offline and syncs when online"
- "Design for intermittent connectivity"
- "Handle synchronization errors gracefully"

# Rule 3: Request Mobile-First Design
All features must be mobile-first:
- Optimize for small screen sizes
- Use touch-friendly interfaces
- Avoid complex navigation menus
- Focus on core functionality first
- Consider limited storage and battery
- Optimize for limited network bandwidth
- Use efficient data compression techniques
- Avoid unnecessary data transfers
- Use lazy loading for images and resources
- Implement pagination for large datasets
- Use client-side rendering for faster loading

Always include in prompts:
- "Design for basic smartphones first"
- "Ensure functionality on small screens"
- "Optimize for touch interactions"
- "Consider offline-first design principles"
- "Optimize for limited storage"
- "Use efficient caching strategies"
- "Avoid storing sensitive user data"
- "Implement secure data encryption"
- "Follow best practices for data privacy"

# Rule 4: Ask for Prioritization
Always ask TRAE to prioritize recommendations:
- "Prioritize by impact vs. implementation effort"
- "What's the 80/20 solution for this feature?"
- "What can I realistically complete this week?"
- "What's the minimum viable version for exhibition?"
- "What are the top 3 features to implement?"

Focus on:
- Core functionality over extras
- User pain points and frustrations
- Exhibition deadline requirements


# Rule 5: Request Code Examples
Always ask for specific code implementations:
- "Show me the exact code to implement this"
- "Provide copy-paste ready code snippets"
- "Include file paths and where to place code"
- "Explain the code in simple terms"
- "Add error handling and validation"
- "Use appropriate data types and formats"

Code quality requirements:
- Use clear and concise variable names
- Add comments to explain complex logic
- Handle edge cases and error conditions
- Follow best practices for security

# Rule 6: Exhibition Preparation Focus
For exhibition-related features:
- "Create a compelling 3-minute demo flow"
- "What will impress potential users and investors?"
- "How can I demonstrate real value for Ethiopian farmers?"
- "What's the exhibition fallback if this doesn't work?"
- "Prepare sample data for realistic demonstration"

Always consider:
- Exhibition audience (farmers, investors, officials)
- Time constraints for demonstration
- Visual impact vs. technical complexity

# Rule 7: Request Testing for Ethiopian Conditions
Always ask for testing guidance:
- "How can I test this with poor connectivity?"
- "What's the best way to test on basic smartphones?"
- "How can I simulate Ethiopian network conditions?"
- "What edge cases should I test for farmers?"
- "How can I test for data privacy?"
- "What security measures are in place?"
- "How can I test for data encryption?"
- "What are the best practices for data privacy?"


Testing requirements:
- Offline functionality verification
- Low-bandwidth performance testing
- Multi-language support validation
- User experience on basic devices
- Security testing for sensitive data
- Compatibility testing with different browsers
- Performance testing on slow networks

# Rule 8: Request Simple Explanations
Always ask for clear explanations:
- "Explain this to me like I'm new to development"
- "Break this down into simple steps"
- "Use analogies related to farming or livestock"
- "Provide a summary of what this code does"
- "Add comments to explain complex logic"
- "Use clear and concise variable names"
- "Handle edge cases and error conditions"
- "Follow best practices for security"
- "Handle errors gracefully"

For technical concepts:
- Request step-by-step instructions
- Ask for visual explanations when helpful
- Request real-world examples from Ethiopian context
- "Explain the importance of this concept in Ethiopian farming" 
- "Provide real-world case studies from Ethiopian farmers"
- "Highlight the challenges they face"
- "Show how this concept solves their problems"
- "Discuss the impact on Ethiopian farmers' livelihoods"
- "Show how this concept can improve Ethiopian farming practices"
- "Discuss the long-term benefits for the Ethiopian farming community"
